# Harvest — Copilot Instructions

You are assisting in a **Flutter e-commerce application** called **Harvest**.
Harvest is a fresh food e-commerce app focused on perishable products (fruits, vegetables, greens) connecting communities directly with local farmers. The goal is to make healthy food easily accessible with a fast and reliable experience.

Generate **high-quality, performant, and maintainable** code that follows the conventions established in this codebase.

## Architecture

### Pattern — Clean Architecture (Feature-First) + SOLID

```
lib/
├── app/                          # App-level setup
│   ├── app_widget.dart           # MaterialApp.router + Theme + BlocProviders
│   ├── assets/
│   │   ├── i18n/                 # Translation JSON files
│   │   └── images/               # Logos, placeholders
│   ├── di/
│   │   └── injection_container.dart  # get_it setup
│   ├── routes/
│   │   ├── app_router.dart       # GoRouter config + auth guard
│   │   └── app_routes.dart       # Route path constants
│   ├── theme/
│   │   ├── app_colors.dart       # Palette tokens
│   │   ├── app_theme.dart        # ThemeData builder
│   │   └── app_typography.dart   # Text style tokens
│   └── widgets/                  # Shared reusable widgets
├── core/                         # Shared across all features
│   ├── errors/
│   │   ├── failures.dart         # Failure sealed class
│   │   └── exceptions.dart       # Custom exceptions
│   ├── extensions/               # BuildContext, String extensions
│   └── utils/                    # Validators, currency formatter
├── features/
│   ├── onboarding/               # Onboarding flow (3 steps)
│   ├── auth/                     # data/domain/presentation (Firebase Auth)
│   ├── home/                     # Categories + featured products
│   ├── search/                   # Search + filters
│   ├── product_details/          # Product detail page
│   ├── cart/                     # Cart management (local state)
│   ├── checkout/                 # Address + mocked payment
│   ├── orders/                   # Order history from Firestore
│   └── profile/                  # User data, menu, sign out
├── gen/                          # Auto-generated (slang)
│   └── i18n/
└── main.dart
```

### Feature Structure (each feature follows this pattern)

```
feature_name/
├── data/
│   ├── datasources/              # Remote/local data sources
│   ├── models/                   # JSON serialization (fromJson/toJson, fromFirestore)
│   └── repositories/             # Repository implementations
├── domain/
│   ├── entities/                 # Pure Dart classes (no external deps)
│   ├── repositories/             # Abstract repository contracts
│   └── usecases/                 # Single-responsibility use cases
└── presentation/
    ├── bloc/ or cubit/           # State management
    ├── pages/                    # Route-level screens
    └── widgets/                  # Feature-specific widgets
```

### Data Flow

```
UI (Page/Widget) → Bloc/Cubit → UseCase → Repository (abstract) → DataSource (Firebase)
```

- Domain entities are plain Dart classes with no external dependencies.
- Models (data layer) contain `fromJson`/`toJson` and `fromFirestore` conversions.
- Repositories are abstract in domain, implemented in data.
- `get_it` for dependency injection (service locator pattern).
- `Either<Failure, T>` from `dartz` for error handling in use cases.

## State Management — flutter_bloc

- **Bloc** for features with complex event-driven logic (Auth, Home, Cart, Checkout, Orders).
- **Cubit** for simpler state (Search, Profile).
- Every Bloc/Cubit state extends `Equatable`.
- States use sealed classes or enums for status (initial, loading, loaded, error).
- Never emit states with raw strings for errors — use `Failure` objects.

---

## Error Handling

- Use cases return `Either<Failure, T>`.
- Failures are a sealed class hierarchy: `ServerFailure`, `AuthFailure`, `CacheFailure`.
- DataSources throw `ServerException`, `AuthException`.
- Repositories catch exceptions and return `Left(Failure)`.
- Blocs/Cubits handle `Either` and emit appropriate states.

## Internationalisation (i18n) — Slang

- Config: `slang.yaml` — base locale `en`, input from `lib/app/assets/i18n/`, output to `lib/gen/i18n/`.
- Source file: `lib/app/assets/i18n/en.i18n.json`.
- Access translations via the global `t` object (e.g., `t.home.title`, `t.cart.empty`).
- After editing JSON files, regenerate with `dart run slang`.
- **Every user-facing string** must come from slang — zero hardcoded strings in widgets.

## Design Principles

- **SOLID** — Single responsibility per class; depend on abstractions.
- **DRY** — Extract shared logic into `core/utils` or shared widgets.
- **KISS** — Favour simplicity. Don't add abstractions unless they reduce complexity.
- **Composition over inheritance** — Prefer composing small widgets.
- **Responsiveness** — Every page must render well on mobile, tablet, and desktop.
- **Performance** — Use `const` constructors, cached images, shimmer loading.

## Coding Conventions

| Rule | Detail |
|---|---|
| **Naming** | `snake_case` files, `PascalCase` classes, `camelCase` members |
| **Imports** | Use package imports (`package:harvest/...`), not relative |
| **`const`** | Apply `const` constructors and `const` values wherever possible |
| **Private fields** | Prefix with `_`; expose only what's needed |
| **File length** | Keep files under 200 lines; split if larger |
| **Widget decomposition** | Extract widgets when they exceed ~50 lines or are reused |
| **Comments** | Only where the *why* isn't obvious; avoid restating the code |
| **Lint rules** | Follow `very_good_analysis`; never add `// ignore` without justification |
| **Error handling** | Always use `Either<Failure, T>` in use cases; never throw from repositories |
| **Bloc naming** | Events: past tense (`ProductsLoaded`); States: noun + status (`HomeState`) |
| **i18n** | All strings via `t.section.key` — never hardcode user-facing text |

## When Adding a New Feature

1. Create the feature folder under `lib/features/` with `data/`, `domain/`, `presentation/` subfolders.
2. Define entities in `domain/entities/` — pure Dart classes.
3. Define the repository contract in `domain/repositories/`.
4. Create use cases in `domain/usecases/` — one per business action.
5. Implement models in `data/models/` with `fromJson`/`toJson`.
6. Implement data sources in `data/datasources/`.
7. Implement the repository in `data/repositories/`.
8. Create Bloc/Cubit in `presentation/bloc/` or `presentation/cubit/`.
9. Build pages in `presentation/pages/` and widgets in `presentation/widgets/`.
10. Register all dependencies in `app/di/injection_container.dart`.
11. Add routes in `app/routes/app_router.dart`.
12. Add all user-facing strings to `en.i18n.json` and run `dart run slang`.
